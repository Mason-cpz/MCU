# Key + LED Framework Spec

这是 `BSP_HAL_APP` 下三个工程的唯一需求来源。
后续只改这一份，三个工程的 `README.md` 只保留入口，不再重复写需求。

## 编译工具

- `D:\6.IED\Keil_v5\UV4\UV4.exe`
- 每个工程使用各自目录下的 `MDK-ARM/*.uvprojx`

## 开发原则

- 目标是做一套可复用、可迁移的通用框架，不是把单一工程修到“能跑就行”。
- 优先兼容 STM32 与国产同构芯片，尤其是 GD32 的 F0/F1/F4 对应系列。
- 厂商相关代码只留在最底层适配层，上层框架不要绑定某一家 HAL 的细节名字。
- 可以大胆创新，但必须保留严谨的验证路径，不能为了新颖牺牲稳定、可读性和可维护性。
- 遇到更好的方案，可以明确反驳现有意见；前提是拿出理由、边界和代价，而不是空口争论。
- 判断方案优先看：正确性 > 可移植性 > 可维护性 > 简洁性 > 个人偏好。

## 进一步要求

- 框架要有清晰分层：板级适配、按键层、LED 层、应用层分开。
- 按键层只输出稳定事件，应用层不要直接处理原始抖动 GPIO。
- LED 层要保持非阻塞，闪烁必须由 tick 或等价机制推进。
- 双色灯不要写死成特殊硬件，底层仍按两路 LED 组合。
- 所有需要时间推进的 API 尽量由调用方传入 `now_ms`，核心层不要直接依赖全局 tick，方便移植和测试。
- 所有板级差异都下沉到最底层适配层，框架核心尽量不改。
- 框架要预留后续扩展：更多按键、更多 LED、长按、双击、连按、更多灯效。
- 代码要尽量表驱动、配置驱动，少全局状态，少魔法数，少把逻辑散在 `main.c`。
- 核心 API 要能跨 STM32 和 GD32 共用思路，方便后续迁移。
- 如果某个新方案更好，可以直接提出替代方案；只要论证充分，就不必拘泥于当前写法。

## 目标

核心任务不是做一个孤立 Demo，而是开发一套可复用的 `按键 + LED` 框架。
Demo 只是验证这个框架是否好用、是否容易扩展。

这个框架要让三个工程都能用同一套思路开发，后续新增按键、LED、灯效或交互时，尽量只改配置和状态机，不重写底层结构。

## 术语

- **稳定边沿事件**：经过消抖后的单次物理 GPIO 电平跳变。按键层只输出这种事件，不把原始抖动电平直接交给 `app`。
- **短按**：按下后在 `SHORT_PRESS_MS`（默认 1000ms）内释放。该语义预留给后续手势扩展；当前 Demo 不要求识别短按，直接用稳定边沿事件验证框架即可。
- **上升沿 / 下降沿**：除非特别说明，均指**物理 GPIO 电平跳变**（0→1 / 1→0），不是逻辑按下/释放。逻辑按下/释放由 `active_low` 配置换算。
- **主循环 tick 周期**：建议 ≤ 10ms；消抖窗口默认 `DEBOUNCE_MS = 20ms`。

## 跨平台范围

- 本次交付**只在 STM32F103RC 上实际编译验证**。
- 对 GD32 的要求是**预留**：厂商相关代码（时钟、GPIO、时基）必须封装在最底层适配层，上层 API 不得直接出现 `HAL_*` / `GPIOA` 等厂商符号。
- 后续迁移 GD32 时，原则上只新增 `platform/gd32fxxx/` 目录，保持 `drivers/app/bsp` 的对外 API 不变；若确有必要调整，也应把差异继续收敛回平台层。

## 框架职责

### 按键框架

- 识别 `KEY1/KEY2/KEY3` 的边沿事件。
- `KEY1` 使用上升沿触发（物理 0→1）。
- `KEY2` 和 `KEY3` 使用下降沿触发（物理 1→0）。
- 必须有消抖，窗口 `DEBOUNCE_MS = 20ms`。
- 对上层输出稳定事件，不让 `app` 直接读原始抖动 GPIO。
- 中断、轮询或混合实现都可以，只要对上层行为一致。

### LED 框架

- 管理 4 个 LED 的独立状态。
- 支持常亮、关闭、闪烁等基本模式。
- 闪烁必须是非阻塞的，由 tick 推进。
- `LED3` 和 `LED4` 作为组合灯使用时，可通过同时点亮表示黄色。
- 双色灯底层仍按两路 LED 处理，不把颜色写死成特殊硬件。

### 应用层

- `app` 只消费按键事件，驱动 LED 状态机。
- `app` 负责 Demo 逻辑，不负责硬件抖动处理。
- 结构应方便后续继续加按键、加 LED、加灯效、加状态。

## Demo 行为（验收参考）

### 上电默认态

- `LED1`：1Hz 心跳闪烁，仅用于上电展示；首次 `KEY1` 有效事件后退出心跳态。
- `LED2`：关闭。
- `LED3 + LED4`：关闭。

### KEY1: PA0 上升沿

控制 `LED1`，按一次有效上升沿后循环：

`OFF -> ON -> BLINK(500/500) -> OFF`

说明：上电心跳不计入循环，首次有效事件先把 `LED1` 置为 `OFF`。

### KEY2: PA1 下降沿

控制 `LED2`，按一次有效下降沿后循环：

`OFF -> ON -> BLINK(200/800) -> BLINK(800/200) -> OFF`

### KEY3: PA4 下降沿

控制双色灯组合，按一次有效下降沿后循环：

`OFF -> GREEN -> RED -> YELLOW -> YELLOW BLINK(500/500) -> OFF`

## 扩展方向

- 后续可以加更多按键和更多 LED。
- 后续可以加长按、双击、连按等按键行为。
- 后续可以加更多灯效，但基础 API 不应频繁推翻。
- 后续实现如果要改结构，应优先动配置、表和状态机，而不是把框架打碎重来。

## 交付标准

- 三个工程都围绕同一套框架目标开发。
- 代码风格允许不同，但可维护性和扩展性要明显优于一把梭。
- 后续新增功能时，只改这份规范即可统一方向。
